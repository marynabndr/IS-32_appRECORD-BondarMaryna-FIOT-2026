```morgan:``` Це "реєстратор" (logger) HTTP-запитів. Вона автоматично відстежує кожен запит, який приходить на ваш сервер (наприклад, GET, POST), і виводить у консоль інформацію про нього: який метод було використано, яка адреса запиту, який статус відповіді (наприклад, 200 — успіх, 404 — не знайдено) та скільки часу це зайняло (мс).

```winston:``` Це потужна система для створення логів. На відміну від console.log, winston дозволяє записувати повідомлення у файли (наприклад, app.log), розділяти їх за рівнем важливості (info, error, debug) та зберігати історію подій, що відбуваються на сервері, навіть якщо ви закрили консоль.

```multer:``` Це мідлвара (middleware) для обробки завантаження файлів. Express.js за замовчуванням не вміє "читати" файли, які надсилає користувач. multer перехоплює файли (картинки, PDF тощо), зберігає їх у вказану папку на сервері та додає інформацію про них у об'єкт req.file.

```pm2:``` Це менеджер процесів. Він потрібен для того, щоб ваш сервер працював "вічно". Якщо у коді станеться помилка і сервер "впаде", PM2 автоматично запустить його знову. Також він дозволяє моніторити навантаження на процесор та пам'ять.

### 3. Файлове логування подій

Інтегруйте Winston і реалізуйте:
- запис логів у файл app.log
- логування рівнів: info, error

Налаштування Winston у ```server.js```

Мета завдання: впровадити надійну систему логування, яка дозволяє зберігати історію подій у структурованому форматі (JSON) та розділяти повідомлення за ступенем важливості (рівнями).

Для реалізації логування було створено об'єкт конфігурації ```logger``` за допомогою методу ```winston.createLogger```. 

Ключові аспекти налаштування:

- Рівні логування ```(level: 'info')```: Ми встановили базовий рівень ```info```. Це означає, що логер буде фіксувати події рівня info, warn та error. Це дозволяє відсіювати зайву технічну інформацію (наприклад, debug) у фінальному файлі.

- Форматування ```(format)```:

```timestamp()``` — автоматично додає до кожного запису точну дату та час події.

```json() ```— перетворює лог у формат об'єкта. Це дозволяє в майбутньому легко аналізувати логи за допомогою автоматизованих систем.

- Транспорти ```(transports)```: Це канали, куди відправляються логи.
```File``` записує всі події у файл ```app.log```, що забезпечує довготривале зберігання історії роботи сервера.
```Console``` дублює повідомлення в термінал у спрощеному вигляді для зручності розробника під час налагодження коду.

```
const winston = require('winston');

// Створюємо конфігурацію логера
const logger = winston.createLogger({
    level: 'info', // Мінімальний рівень логування
    format: winston.format.combine(
        winston.format.timestamp(), // Додає час події
        winston.format.json()      // Формат запису — JSON (зручно для аналізу)
    ),
    transports: [
        // Записувати все у файл app.log
        new winston.transports.File({ filename: 'app.log' }),
        // Також дублювати вивід у консоль (опціонально, для зручності розробки)
        new winston.transports.Console({
            format: winston.format.simple()
        })
    ],
});
```
Інтегрували логер безпосередньо в життєвий цикл запуску додатка, замінивши стандартні методи ```console```:

- Логування успішних подій ```(logger.info)```: Ми використовуємо цей метод для фіксації критично важливих етапів: успішної синхронізації бази даних та фактичного запуску сервера на певному порту. Запис у файлі підтверджує, що додаток працює стабільно.

- Логування помилок ```(logger.error)```: У блоці catch, який перехоплює виняткові ситуації (наприклад, якщо база даних відключена або порт зайнятий), ми викликаємо ```logger.error```. Це записує у файл не просто повідомлення "Помилка", а детальний текст винятку ```(error.message)```, що значно полегшує пошук та виправлення причин збою.


```
async function startServer() {
  try {
    await sequelize.sync({ alter: true }); 
    
    // Замість console.log використовуємо logger.info
    logger.info("Базу даних синхронізовано успішно");
    console.log("Базу даних синхронізовано"); 

    app.listen(PORT, () => {
        // Логуємо запуск сервера у файл
        logger.info(`Сервер запущено на http://localhost:${PORT}`);
        console.log(`Сервер: http://localhost:${PORT}`);
    });

  } catch (error) {
    // Замість console.error використовуємо logger.error
    logger.error(`Помилка старту сервера: ${error.message}`);
    console.error("Помилка старту:", error);
  }
}
```

### 4. Обробка помилок

Реалізуйте middleware для обробки помилок:
- логування помилок через Winston
- повернення користувачу JSON-відповіді з кодом помилки

Мета завдання: створити єдиний механізм обробки виняткових ситуацій, щоб сервер не "падав" при виникненні помилок, а користувач отримував зрозумілу відповідь у форматі JSON.

Замість простого виводу в консоль, мідлвара використовує створений раніше ```logger.error```. Крім самого повідомлення про помилку (err.message), ми фіксуємо:
- Stack trace: дозволяє розробнику точно побачити ланцюжок викликів функцій, що призвели до збою.
- Контекст запиту: метод (GET/POST) та URL, на якому сталася помилка. Це критично важливо для відтворення багів.

```
// Мідлвара для обробки помилок 
app.use((err, req, res, next) => {
    // Логування помилки через Winston у файл app.log
    logger.error({
        message: err.message,
        stack: err.stack, // додає інформацію про те, в якому рядку коду сталася помилка
        url: req.originalUrl,
        method: req.method,
        ip: req.ip
    });

    // Повернення користувачу JSON-відповіді
    const statusCode = err.status || 500; // якщо код не вказано, використовуємо 500 (Internal Server Error)
    res.status(statusCode).json({
        success: false,
        status: statusCode,
        message: "На сервері сталася помилка",
        error: err.message
    });
});
```

Перевірка (тестовий маршрут)

```
app.get('/api/test-error', (req, res) => {
    throw new Error("Тестова помилка для перевірки Winston!");
});
```

### 5. Завантаження одного файлу ```post(upload)```

Реалізуйте endpoint /upload, який дозволяє завантажити один файл за допомогою Multer.

Мета завдання: Навчити сервер приймати бінарні дані (файли) від клієнта за допомогою стандартної бібліотеки Multer.

- Бібліотека ```Multer```: оскільки Express за замовчуванням не може розібрати формат даних ```multipart/form-data``` (в якому надсилаються файли), ми використовуємо Multer як посередника.

- ```upload.single('file')```:
Метод ```.single``` вказує, що ми очікуємо лише один файл.
Параметр ```'file'``` — це ім'я поля (Key), яке має використовувати клієнт.

- Об'єкт ```req.file```: Після того, як Multer обробить запит, він додає до об'єкта запиту нове поле file. Воно містить метадані: назву, розширення, розмір у байтах та тимчасовий шлях до збереженого файлу на диску.

1. Запит POST на адресу http://localhost:3000/upload.
2. Вкладка Body -> оберіть Form-data.
3. У полі Key вводимо file
4. Обераємо тип File (замість Text) і будь-який документ на комп'ютері.
5. Send.

```
const multer = require('multer');
const upload = multer({ dest: 'uploads/' }); // файли потраплятимуть у папку uploads

// Завдання 5: Ендпоінт для завантаження одного файлу
app.post('/upload', upload.single('file'), (req, res) => {
    // Перевірка чи файл дійсно прийшов
    if (!req.file) {
        return res.status(400).json({ message: "Будь ласка, оберіть файл для завантаження" });
    }

    // Відповідь користувачу з даними про файл
    res.json({
        message: "Файл завантажено успішно",
        fileInfo: {
            originalName: req.file.originalname,
            size: req.file.size,
            path: req.file.path
        }
    });
});
```

### 6. Завантаження кількох файлів ```post(upload-multiple)```

Розширте попереднє завдання:

- підтримка завантаження кількох файлів
- збереження їх у папку /uploads

Мета завдання: розширити функціонал сервера для обробки масивів даних (кількох файлів одночасно) та забезпечити їхнє впорядковане зберігання.

- ```upload.array('files', 5)```: Цей метод вказує Multer, що він має шукати масив файлів під ключем files. Другий параметр (5) встановлює ліміт на кількість файлів за один запит, що захищає сервер від спаму.

- Об'єкт ```req.files```: Для кількох файлів Multer використовує множину (files), яка є масивом об'єктів. Кожен об'єкт містить деталі окремого файлу.

- Контроль імен: Використання ```Date.now()``` у назві файлу вирішує проблему "однакових імен". 

- Папка ```/uploads```: Всі файли автоматично потрапляють у вказану директорію, що дозволяє легко керувати статичним контентом сервера.


1. Запит POST на адресу http://localhost:3000/upload-multiple.
2. Створюємо декілька рядків Вкладка Body -> оберіть Form-data.
3. У полі Key вводимо files у кожному рядку.
4. Обераємо тип File (замість Text) і будь-які документи на комп'ютері.
5. Send.

```
// Завдання 6: Завантаження кількох файлів (до 5 штук)
app.post('/upload-multiple', upload.array('files', 5), (req, res) => {
    // Перевірка, чи були надіслані файли
    if (!req.files || req.files.length === 0) {
        return res.status(400).json({ message: "Файли не завантажено" });
    }

    // Повертаємо список завантажених файлів
    res.json({
        message: `Успішно завантажено файлів: ${req.files.length}`,
        files: req.files.map(f => ({
            name: f.filename,
            size: f.size
        }))
    });
});
```

### 7. Валідація файлів ```post(upload-secure)```

Додайте перевірки:
- дозволені типи файлів (наприклад: jpg, png, pdf)
- обмеження розміру файлу

Мета завдання: pабезпечити технічний контроль над даними, що надходять від користувачів, для захисту сервера від шкідливих файлів та переповнення дискового простору.

- Розділення логіки: Створення окремого об'єкта ```uploadValidated``` дозволяє застосовувати суворі правила безпеки (валідацію) лише для конкретних маршрутів (наприклад, завантаження документів), залишаючи інші маршрути відкритими.

- Перевірка типів (MIME-types): реалізовано фільтрацію за MIME-типом ```(image/jpeg, image/png, application/pdf)```. Це надійніший спосіб, ніж перевірка розширення файлу, оскільки він аналізує фактичний вміст файлу, захищаючи сервер від завантаження шкідливих скриптів.

- Обмеження ресурсів (limits): Встановлення ліміту ```fileSize``` на рівні 1 МБ запобігає атакам типу "відмова в обслуговуванні" (DoS), коли зловмисник намагається переповнити дисковий простір сервера занадто великими файлами.

- Користувацький досвід: Завдяки інтеграції з раніше створеною мідлварою помилок, користувач у разі помилки валідації отримає не просто збій, а структуровану JSON-відповідь із причиною відмови.


```
// ЗАВДАННЯ 7: Валідація 
//  об'єкт зберігання для валідації
const storageSecure = multer.diskStorage({
    destination: (req, file, cb) => {
        cb(null, 'uploads/');
    },
    filename: (req, file, cb) => {
        cb(null, 'secure-' + Date.now() + '-' + file.originalname);
    }
});

// Функція фільтрації типів (JPG, PNG, PDF)
const fileFilterSecure = (req, file, cb) => {
    const allowedTypes = ['image/jpeg', 'image/png', 'application/pdf'];
    if (allowedTypes.includes(file.mimetype)) {
        cb(null, true); 
    } else {
        cb(new Error('Недопустимий тип файлу! Дозволено лише JPG, PNG та PDF.'), false);
    }
};

// створення об'єкта Multer
const uploadValidated = multer({ 
    storage: storageSecure, 
    limits: { 
        fileSize: 1 * 1024 * 1024 // Ліміт: 1 МБ
    },
    fileFilter: fileFilterSecure
});

// ендпоінт для перевірки
app.post('/upload-secure', uploadValidated.single('file'), (req, res) => {
    res.json({
        message: "Файл пройшов валідацію і завантажений успішно",
        file: req.file
    });
});
```

### 8. Моніторинг стану сервера ```get(status)```

Створіть endpoint /status, який повертає:
- uptime сервера
- використання пам’яті (process.memoryUsage())

Мета: реалізувати перевірку сервера. Це дозволяє розробнику або системному адміністратору миттєво дізнатися:

- Чи взагалі працює сервер.
- Як довго він працює без перезавантажень ```(uptime)```.
- Скільки ресурсів він споживає ```(memoryUsage)```. Це критично для виявлення "витоків пам'яті", коли програма поступово "з’їдає" всю оперативну пам'ять і в результаті "падає".

```process.uptime()``` — стандартна функція Node.js, яка повертає час роботи процесу в секундах. Ми перетворюємо їх у хвилини та секунди для зручності читання.

```process.memoryUsage()``` — повертає об'єкт зі статистикою пам'яті у байтах.

```
// Завдання 8: Моніторинг стану сервера
app.get('/status', (req, res) => {
    const memoryUsage = process.memoryUsage();
    const uptime = process.uptime(); // Час роботи сервера в секундах
    
    res.json({
        status: "Server is healthy",
        uptime: `${Math.floor(uptime / 60)} minutes ${Math.floor(uptime % 60)} seconds`,
        memoryUsage: {
            rss: `${(memoryUsage.rss / 1024 / 1024).toFixed(2)} MB`, // Загальна пам'ять
            heapTotal: `${(memoryUsage.heapTotal / 1024 / 1024).toFixed(2)} MB`, // Виділено пам'яті
            heapUsed: `${(memoryUsage.heapUsed / 1024 / 1024).toFixed(2)} MB`,  // Використано пам'яті
        },
        nodeVersion: process.version,
        platform: process.platform
    });
});
```
Ми бачимо стан пам'яті, час роботи та версію Node.js.

### 9. Вимірювання часу відповіді

Реалізуйте middleware, який:
- вимірює час обробки кожного запиту
- записує результат у лог

```
// Завдання 9: Middleware для вимірювання часу відповіді
app.use((req, res, next) => {
    // Фіксуємо час початку запиту
    const start = Date.now();

    // Чекаємо, поки сервер повністю закінчить відправку відповіді клієнту
    res.on('finish', () => {
        // Обчислюємо різницю 
        const duration = Date.now() - start;
        const logMessage = `${req.method} ${req.originalUrl} - ${duration}ms`;

        // Записуємо результат у лог (використовуємо Winston logger)
        logger.info({
            type: 'performance',
            method: req.method,
            url: req.originalUrl,
            duration: `${duration}ms`
        });

        console.log(`[PERFORMANCE] ${logMessage}`);
    });

    next(); 
});
```

### 10. Інтеграція менеджера процесів

Запустіть застосунок через PM2 та:
- перевірте автоматичний рестарт
- перегляньте логи через PM2

Запуск сервера через PM2 та таблиця процесів

```
pm2 start server.js --name "my-lab-server"
```

- Daemonized: Ми бачимо, що PM2 запустив фоновий процес (daemon), який буде стежити за додатком.

- Статус додатка: У таблиці відображається наш сервер під назвою my-lab-server.

- Метрики: Статус online підтверджує, що сервер працює. Також зафіксовано початкове споживання ресурсів: CPU (1.6%) та Memory (78.6mb). Це початкова точка моніторингу.

= Metadata: Статистична інформація. Найважливіше тут — показник Restarts: 28, який підтверджує, що PM2 не дає серверу остаточно "впасти", перезапускаючи його після кожної помилки.

### Висновки

Під час виконання лабораторної роботи №4 я ознайомилася з розширеними можливостями Node.js для створення стабільних серверних застосунків. У ході роботи було успішно реалізовано:

1. Систему логування: поєднання інструментів Morgan (для HTTP-запитів) та Winston (для запису критичних подій та продуктивності у файл), що дозволяє ефективно відстежувати роботу сервера.
2. Функціонал завантаження файлів: за допомогою бібліотеки Multer налаштовано приймання файлів із суворою валідацією їхнього типу (JPG, PNG, PDF) та розміру (до 1 МБ), що підвищує безпеку додатка.
3. Моніторинг продуктивності: реалізовано ендпоінт для перевірки стану ресурсів (uptime, memoryUsage) та проведено інтеграцію з менеджером процесів PM2.

Використання PM2 дозволило забезпечити високу доступність сервера завдяки функції автоматичного перезапуску при збоях та наочному моніторингу навантаження на систему в реальному часі. Отримані навички є фундаментом для розробки відмовостійких систем, готових до промислової експлуатації (production).
