Валідація:
- isEmailConfirmed: Статус підтвердження пошти (для безпеки та перевірки реальності користувача).

Службові поля:

```
const User = sequelize.define('User', {
  name: { type: DataTypes.STRING, allowNull: false },
  email: { type: DataTypes.STRING, allowNull: false, unique: true },
  password: { type: DataTypes.STRING, allowNull: true }, // allowNull: true для Google OAuth
  role: { 
    type: DataTypes.ENUM('user', 'admin'), 
    defaultValue: 'user' 
  },
  isEmailConfirmed: { type: DataTypes.BOOLEAN, defaultValue: false },

  resetPasswordToken: { type: DataTypes.STRING, allowNull: true }
}, { timestamps: true }); 

module.exports = User;
```

### Створення Middleware для перевірки токена та ролей (```middleware/auth.js```)

```
- Завдання 15 (Middleware для токена): Вся логіка розшифрування JWT (jwt.verify).
- Завдання 4 (Захищений маршрут): Цей файл сам по собі є механізмом, який робить будь-який маршрут захищеним.
```

Middleware (проміжне ПЗ) — це функції, які мають доступ до об’єктів запиту (req), відповіді (res) та наступної функції в циклі (next). Вони виконують роль фільтрів або обробників. 

Middleware для Express відповідає за перевірку доступу до захищених ресурсів. Він реалізує створення захищеного маршруту за допомогою JWT-токена.

Імпорт та налаштування:

- ```const jwt = require('jsonwebtoken');```: підключення бібліотеки для роботи з JSON Web Tokens.

- ```ACCESS_SECRET```: секретний ключ, який використовується для перевірки того, що токен був підписаний саме вашим сервером і не був підроблений.

Логіка перевірки (тіло функції):

Функція приймає три аргументи: req (запит), res (відповідь) та next (функція для переходу до наступного обробника).

- Отримання токена:
```const token = req.headers.authorization.split(' ')[ 1 ]```;
Код шукає токен у заголовках HTTP-запиту. Зазвичай він передається у форматі `Bearer <token>`, тому метод split(' ')[ 1 ] забирає саме другу частину (сам токен).

- Перевірка валідності:
```const decoded = jwt.verify(token, ACCESS_SECRET)```;
Бібліотека JWT перевіряє токен. Якщо термін дії токена закінчився або підпис невірний, код перейде в блок catch.

- Збереження даних:
```req.user = decoded```;
Дані користувача (наприклад, його id або email), що були зашифровані в токені, додаються безпосередньо в об'єкт запиту. Це дозволяє наступним функціям (наприклад, профілю) знати, хто саме робить запит.

- Перехід далі:
```next()```;
Якщо токен вірний, викликається функція next(), яка дозволяє серверу продовжити виконання запиту та повернути дані клієнту.

Обробка помилок:

Якщо токена немає або він невірний, спрацьовує блок ```catch (e)```:
- Сервер повертає HTTP-код 401 (Unauthorized).
- Клієнт отримує JSON-повідомлення про те, що доступ заборонено.

```
const jwt = require('jsonwebtoken');
const ACCESS_SECRET = "secret_access_key"; 

module.exports = (req, res, next) => {
    try {
        const token = req.headers.authorization.split(' ')[1];
        if (!token) throw new Error();
        
        const decoded = jwt.verify(token, ACCESS_SECRET);
        req.user = decoded; 
        next();
    } catch (e) {
        res.status(401).json({ message: "Не авторизований (немає токена)" });
    }
};
```

Завдання 14: Обмеження спроб входу (Rate Limit)

Захист від brute-force атак (автоматичного підбору паролів). Якщо зловмисник спробує підібрати пароль більше 5 разів за 15 хвилин, сервер тимчасово заблокує його запити.
```
const loginLimiter = rateLimit({
    windowMs: 15 * 60 * 1000, 
    max: 5, 
    message: { message: "Забагато невдалих спроб. Спробуйте через 15 хвилин." }
});
```

```
if (password !== confirmPassword) {
```

Хешування паролів (bcryptjs)

Це стандарт безпеки №1. Ми не зберігаємо паролі в чистому вигляді. У базі зберігається лише «відбиток» (хеш), за яким неможливо дізнатися оригінальний пароль.

```
// При реєстрації (Завдання 11)
const hashedPassword = await bcrypt.hash(password, 10);

// При вході (Завдання 2)
const isMatch = await bcrypt.compare(password, user.password);
```

Завдання 2 та 12: Реєстрація, Вхід та Refresh Token

Створення нових користувачів та видача токенів для підтримки сесії. accessToken — для доступу, refreshToken — для того, щоб користувач не перелогінювався кожні 15 хвилин.

```
app.post('/api/register', async (req, res) => { ... });

app.post('/api/login', loginLimiter, async (req, res) => {
    // Генерація токенів
    const accessToken = jwt.sign({ id: user.id, role: user.role }, ACCESS_SECRET, { expiresIn: '15m' });
    const refreshToken = jwt.sign({ id: user.id }, REFRESH_SECRET, { expiresIn: '7d' });
    
    // Збереження в базу (Пункт 12)
    user.refreshToken = refreshToken;
    await user.save();
});
```

Завдання 9: Реалізація Logout (Вихід)

Безпечне завершення сесії. Сервер видаляє refreshToken з бази даних, щоб старий токен більше не міг бути використаний.

```
app.post('/api/logout', authMiddleware, async (req, res) => {

});
```

## Відповіді на контрольні запитання
**1. Що таке REST API та які його основні принципи** 

REST (Representational State Transfer) — це архітектурний стиль взаємодії компонентів розподіленого застосунку в мережі. Це не протокол, а набір правил, за якими клієнт і сервер обмінюються даними.

Основні принципи:

- Клієнт-серверна модель: Розмежування інтерфейсу користувача та логіки збереження даних.

- Stateless (Відсутність стану): Кожен запит містить всю необхідну інформацію для його обробки.

- Кешування: Відповіді повинні мати позначку, чи можна їх зберігати локально.

- Єдиний інтерфейс: Стандартні методи взаємодії (наприклад, HTTP-методи).

- Багаторівнева система: Клієнт не знає, чи він підключений безпосередньо до сервера, чи до посередника (проксі).

**2. Які HTTP-методи використовуються в REST API і для чого вони призначені?**

- GET: отримання даних.
- POST: створення нового ресурсу.
- PUT: повне оновлення ресурсу.
- PATCH: часткове оновлення.
- DELETE: видалення ресурсу.

**3. Що таке клієнт-серверна архітектура?**

Це концепція, де мережа розділена на дві активні ролі:

- Клієнт (Client): Надсилає запити (браузер, мобільний застосунок). Він відповідає за візуалізацію та взаємодію з користувачем.

- Сервер (Server): Приймає запити, обробляє логіку, працює з базою даних і повертає результат.
Це дозволяє розробляти фронтенд та бекенд незалежно один від одного.

**4. Що означає принцип Stateless у REST?**

Stateless означає, що сервер не зберігає інформацію про попередні запити клієнта (стан сесії). Кожен запит від клієнта "ізольований" і має містити токен доступу або параметри, щоб сервер міг зрозуміти, хто робить запит і що саме потрібно зробити. Це спрощує масштабування системи.

**5. Для чого використовується фреймворк Express у Node.js?**

Express — це мінімалістичний та гнучкий вебфреймворк для Node.js. Його використовують для:

- Створення серверної частини (Backend).

- Налаштування маршрутизації (обробка шляхів на кшталт /api/users).

- Керування проміжним ПЗ (middleware).

- Спрощення роботи з HTTP-запитами та відповідями.

**6. Що таке реєстрація користувача в інформаційній системі?**

Це процес створення облікового запису в системі. Користувач вводить свої дані (логін, пароль, email), які після перевірки зберігаються в базі даних для подальшого впізнання системою.

**7. Що таке авторизація та чим вона відрізняється від автентифікації?**

- Автентифікація: перевірка, чи є користувач тим, за кого себе видає (логін/пароль).
- Авторизація: перевірка прав доступу користувача до конкретних ресурсів (чи може він видаляти дані).

**8. Для чого використовується JWT (JSON Web Token)?**

JSON Web Token (JWT) використовується для безпечної передачі інформації між сторонами. У REST API він найчастіше слугує інструментом для авторизації. Після входу сервер видає клієнту токен, який той додає до кожного наступного запиту, щоб довести свої права доступу.

**9. З яких частин складається JWT-токен?**

Токен складається з трьох частин, розділених крапками (header.payload.signature):

- Header (Заголовок): Тип токена та алгоритм шифрування (наприклад, HS256).

- Payload (Корисне навантаження): Дані користувача (id, роль, термін дії токена).

- Signature (Підпис): Створюється шляхом хешування перших двох частин із секретним ключем сервера для перевірки цілісності.

**10. Для чого потрібно хешування паролів і яка бібліотека використовується в
роботі?**

Зберігати паролі у відкритому вигляді (текстом) небезпечно. Хешування перетворює пароль на незворотний рядок символів. Навіть якщо базу даних вкрадуть, зловмисники не дізнаються справжні паролі.
У Node.js для цього найчастіше використовується бібліотека bcrypt.

**11. Що таке валідація даних і навіщо вона потрібна?**

Це перевірка вхідних даних на коректність (формат email, довжина пароля) перед їх обробкою, щоб уникнути помилок та атак.

**12. Які основні HTTP-коди стану використовуються для обробки помилок?**

- 200 OK: Запит успішний.
- 201 Created: Ресурс успішно створено.
- 400: Bad Request (помилка клієнта).
- 401: Unauthorized (не авторизований).
- 403: Forbidden (немає прав).
- 404: Not Found (не знайдено).
- 500: Internal Server Error (помилка сервера).

**13. Що таке middleware та яку роль він виконує в Express?**

Middleware (проміжне ПЗ) — це функції, які мають доступ до об'єктів запиту (req), відповіді (res) та наступної функції в циклі (next).
Вони виконують роль фільтрів або обробників, наприклад:

- Логування запитів.
- Перевірка JWT-токена перед допуском до маршруту.
- Парсинг JSON-даних із тіла запиту.

**14. Для чого потрібен захищений маршрут у REST API (Protected Route)?**

Це шлях в API, доступ до якого обмежений. Щоб отримати відповідь від такого маршруту, клієнт обов'язково має надати валідний доказ авторизації (наприклад, JWT у заголовку запиту). Без нього сервер поверне помилку 401. (Щоб обмежити доступ до конфіденційних даних або функцій лише для авторизованих користувачів.)

**15. Що таке роль користувача (user/admin) і як вона впливає на доступ до ресурсів?**

Це атрибут, що визначає рівень повноважень. 
Ролі визначають рівень привілеїв:

- User: Може переглядати свої дані, редагувати свій профіль.

- Admin: Може видаляти будь-яких користувачів, бачити статистику, змінювати налаштування системи.
Це реалізується через перевірку поля role у JWT або базі даних під час обробки запиту.
